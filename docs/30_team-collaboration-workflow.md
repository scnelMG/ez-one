# 30. 2인 팀 협업 규칙

이 문서는 EZ One을 2명이 함께 개발할 때의 작업 약속을 정의한다.

`CONTRIBUTING.md`, `AGENTS.md`, `docs/23_traceability.md`와 함께 사용한다.

## 팀 운영 방식

역할은 고정하지 않고, 기능 단위로 담당자를 정한다.

| 영역 | 주 담당 | 상호 리뷰 |
| --- | --- | --- |
| 백엔드/API/DB | 백엔드 중심 팀원 | 프론트 중심 팀원 |
| 프론트/확장 프로그램 | 프론트 중심 팀원 | 백엔드 중심 팀원 |
| API 계약 | 두 명 모두 | 두 명 모두 |
| 요구사항 범위 | 두 명 모두 | 두 명 모두 |

담당자는 브랜치를 완료 지점까지 끌고 간다. 리뷰어는 요구사항 범위, API 적합성, P1/P2 경계, 검증 결과를 확인한다.

## 브랜치 기준

| 브랜치 | 목적 | 규칙 |
| --- | --- | --- |
| `main` | 안정적인 기준 브랜치 | 빌드 가능하고 리뷰 가능한 상태 유지 |
| `feature/<p1-slice>` | 기능 단위 end-to-end 작업 | 프론트/백엔드 변경이 같은 기능에 속할 때 사용 |
| `feature/<p1-slice>-api` | 백엔드/API 하위 작업 | 백엔드가 독립적으로 진행될 때 사용 |
| `feature/<p1-slice>-ui` | 프론트 하위 작업 | UI가 독립적으로 진행될 때 사용 |
| `docs/<topic>` | 문서/운영 규칙 변경 | 제품 동작 변경과 분리 |

한 명이 기능을 맡아 진행할 때는 기능 브랜치를 우선 사용한다. 두 명이 동시에 작업해야 할 때만 `-api`, `-ui` 브랜치로 나눈다.

## 현재 P1 구현 순서

| 순서 | 브랜치 | 요구사항 ID |
| --- | --- | --- |
| 1 | `feature/auth-google` | `AUTH-001` |
| 2 | `feature/onboarding-profile` | `ONB-001`, `ONB-002` |
| 3 | `feature/basket-job` | `JOB-001`, `JOB-002`, `COMMON-001` |
| 4 | `feature/workspace` | `WS-001`, `WS-002`, `WS-003`, `REF-001` |
| 5 | `feature/document-profile` | `PROFILE-001` |
| 6 | `feature/extension-job-save` | `EXT-001`, `JOB-001` |
| 7 | `feature/notion-job-only` | `NOTION-001` |

명시적인 승인과 문서 갱신 없이 P2 동작을 활성화하지 않는다.

## 기능 시작 체크리스트

코드를 쓰기 전에 아래 항목을 합의한다.

```text
요구사항 ID:
P1/P2/IA 범위:
담당자:
리뷰어:
프론트 라우트/화면:
API 엔드포인트:
요청 DTO:
응답 DTO:
오류 케이스:
소유권 검증 규칙:
갱신할 문서:
실행할 테스트:
```

프론트와 백엔드가 함께 바뀌는 기능은 API 계약을 먼저 정한다. 프론트에서 임시 정적 데이터를 쓰더라도 합의한 응답 형태와 맞아야 한다.

## 데일리 싱크

짧게 공유한다.

```text
어제 한 일:
오늘 할 일:
막힌 점:
상대가 알아야 할 API/DB/화면 변경:
오늘 머지 가능한가:
```

## PR 규칙

PR은 한 번에 리뷰할 수 있을 만큼 작게 만든다.

좋은 PR 범위:

- `AUTH-001` Google 로그인 콜백 엔드포인트
- `JOB-001` 계약 형태의 정적 데이터를 쓰는 공고함 목록 UI
- `PROFILE-001` 서류 정보 섹션 폼
- 협업 규칙 또는 API 계약 문서 갱신

피해야 할 PR 범위:

- 관련 없는 요구사항 ID를 한 PR에 섞기
- P1 동작과 P2 placeholder를 함께 활성화하기
- API 계약과 맞지 않는 UI 변경
- 내부 DB row를 그대로 노출하는 백엔드 응답

## 리뷰 책임

리뷰어는 아래를 확인한다.

- 요구사항 ID와 범위가 명확한가
- P2 또는 IA-only 동작이 활성화되지 않았는가
- API 요청/응답 이름이 문서와 맞는가
- 사용자 소유 데이터에 소유권 검증이 있는가
- 외부 연동 실패가 핵심 저장 흐름을 롤백하지 않는가
- API 기반 UI가 loading, empty, error, unauthorized, forbidden 상태를 다루는가
- 테스트와 검증 명령이 적혀 있는가
- secrets, 실제 토큰, OAuth 코드, 개인정보가 포함되지 않았는가

## 머지 기준

아래 조건을 만족할 때만 머지한다.

- PR 범위가 명확하다.
- 필요한 문서가 갱신됐다.
- 관련 검증이 통과했거나, 생략한 테스트와 이유가 명시됐다.
- 리뷰어가 승인했다.
- `main`이 여전히 올바른 기준 브랜치다.

머지 후에는 의존하는 기능 브랜치를 `main` 기준으로 갱신한 뒤 계속 작업한다.

## 에이전트 사용 방식

프로젝트 에이전트는 담당자가 아니라 보조자다.

| 에이전트 | 사용 방식 |
| --- | --- |
| `explorer` | 구현 전에 문서/코드를 읽고 근거를 요약한다 |
| `implementer` | 승인된 P1 슬라이스에 대해 범위가 좁은 변경을 수행한다 |
| `reviewer` | diff, 범위, 테스트, P1/P2 경계를 읽기 전용으로 검토한다 |

최종 범위와 완료 판정은 루트 Codex 또는 사람 팀이 결정한다.
# 2026-06-04 커밋 메시지 규칙

커밋 메시지는 한글로 작성한다.

규칙:

- 형식은 `<유형>(<범위>): <한글 요약>`을 사용한다.
- 유형은 `feat`, `fix`, `docs`, `refactor`, `test`, `chore` 중 하나를 사용한다.
- 범위는 `auth`, `onboarding`, `dashboard`, `basket`, `workspace`, `references`, `document-profile`, `recommendations`, `notion`, `extension`, `infra`, `docs`처럼 작업 영역을 짧게 쓴다.
- 요약은 팀원이 히스토리만 보고 변경 의도를 이해할 수 있게 한국어로 쓴다.
- 이미 원격에 올라간 커밋은 히스토리 보호를 위해 메시지만 바꾸려고 강제 푸시하지 않는다.

예시:

```text
docs(readme): 루트 README를 제품 소개 중심으로 정리
feat(auth): Google 로그인 성공 흐름 구현
test(router): P2 전용 라우트 미등록 검증 추가
```
