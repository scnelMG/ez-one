# 03. 사용자 리서치 / 경쟁 서비스 분석

기준 원본: Notion `03. 사용자 리서치 / 경쟁 서비스 분석`

## 목적

이 문서는 넓은 시장조사가 아니라 EZ-ONE의 문제 정의를 뒷받침하는 업무 흐름 근거를 정리한다. 기존 도구가 해결하는 영역, 남는 공백, EZ-ONE 요구사항이 메우는 지점을 확인한다.

## 검증할 가설

취업준비생은 공고 저장, 지원 상태, 자소서 작성, 참고자료, 서류 입력 정보를 한 공고 단위로 연결해 관리할 필요가 있다.

## 사용자 문제

| 문제 | 흐름상 근거 | 연결 요구사항 |
| --- | --- | --- |
| 공고 저장과 작성 공간이 분리된다. | 북마크한 공고가 곧바로 작성 워크스페이스가 되지 않는다. | `JOB-001`, `JOB-002`, `WS-001` |
| 우선순위 판단이 어렵다. | 마감일, 지원 상태, 작성 진행도가 여러 도구에 흩어진다. | `DASH-001`, `JOB-001` |
| 참고자료가 초안과 분리된다. | JD, 뉴스, 기업 메모, 프롬프트, 초안이 별도 페이지나 파일에 남는다. | `REF-001`, `REF-002`, `WS-002` |
| 서류 입력 정보를 반복 입력한다. | 학력, 경력, 프로젝트, 자격, 수상/활동 정보가 지원서마다 반복된다. | `PROFILE-001`, `WS-003` |
| 외부 연동은 유용하지만 불안정하다. | 사이트 DOM 변경, Notion 동기화 실패, 외부 인증 실패가 발생할 수 있다. | `EXT-001`, `NOTION-001` |

## 기존 도구 비교

| 기준 | 채용 플랫폼 | Jasoseol.com | Notion/스프레드시트 | EZ-ONE |
| --- | --- | --- | --- | --- |
| 공고 탐색 | 강함 | 지원 중심 탐색에 강함 | 수동 입력 | 저장/추천 공고에서 시작 |
| 공고 저장 | 북마크 중심 | 지원 흐름 중심 | 직접 복사 | 장바구니와 워크스페이스 자동 생성 |
| 상태/마감 확인 | 플랫폼 내부에서 가능 | 지원 흐름 내 가능 | 사용자가 직접 구성 | 대시보드와 장바구니 필터 |
| 자소서 작성 맥락 | 공고 저장과 분리되는 경우가 많음 | 일부 지원 | 자유롭지만 수동 연결 | 공고별 워크스페이스에 연결 |
| 참고자료 관리 | 핵심 기능 아님 | 제한적 | 자유롭지만 분산 가능 | 참고자료 보드와 사이드 패널 |
| 서류 입력 정보 재사용 | 플랫폼별로 분리 | 서비스 내부에 한정 | 직접 복사 | 사용자 단위 문서 프로필 |
| 외부 동기화 | 플랫폼 silo | 플랫폼 silo | 워크스페이스 내부 | MVP는 job-only Notion sync |

## 제품 공백

기존 도구는 각 지점에서는 유용하지만 저장 공고, 마감일, 지원 상태, 자소서 초안, 참고자료, 재사용 정보를 한 공고 단위로 연결하는 층이 약하다. EZ-ONE은 이 연결층에 집중한다.

## 차별화 결정

| 차별점 | 제품 결정 |
| --- | --- |
| 공고별 워크스페이스 | 공고 저장 시 워크스페이스를 자동 생성한다. |
| 장바구니 기반 준비 큐 | 대시보드 카드와 장바구니 필터로 다음 행동을 결정한다. |
| 참고자료 사이드 패널 | JD, 뉴스, DART, 인재상, 메모, 커스텀 보드를 작성 중 열람한다. |
| 서류 입력 정보 재사용 | 사용자 단위로 저장한 프로필 정보를 워크스페이스 기본값에 반영한다. |
| 외부 실패 격리 | Notion/Extension 실패를 핵심 DB 저장과 분리한다. |

## 범위 결정

- P1은 `공고 저장 -> 장바구니 -> 워크스페이스 -> 수동 참고자료/자소서 작성 -> 서류 입력 정보 재사용 -> JOB_ONLY Notion sync`에 집중한다.
- P2는 알림, 과거 지원 내역, Mattermost 후보화, 자동 참고자료 수집, 서류 자동 입력 보조, Notion scope 확장으로 분리한다.
- P1에서 EZ-ONE은 풀스택 채용 탐색 플랫폼이 아니라, 선택한 공고를 준비 가능한 상태로 연결하는 도구다.
