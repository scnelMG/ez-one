# 01. 문제 정의 및 검증 문서

Source of truth: Notion `01. 문제 정의 및 검증 문서`

## 결론

취업준비생은 관심 공고, 마감일, 자기소개서 문항, 참고자료, 본인 경험을 여러 도구에 나누어 관리한다. 이 때문에 어떤 공고부터 준비해야 하는지 판단하기 어렵고, 공고별 작성 맥락을 반복해서 다시 구성한다.

EZ One은 `공고 저장 -> 장바구니 -> 지원 워크스페이스 -> 참고자료/자소서 작성 -> 서류 입력 정보 재사용` 흐름을 하나로 묶어 지원 준비의 분산 문제를 줄인다.

## 대상 사용자

| 사용자 | 현재 상황 | 핵심 니즈 |
| --- | --- | --- |
| 취업준비생 | 여러 기업에 동시에 지원하며 공고, 마감일, 자소서 문항을 관리한다. | 공고별 준비 상태와 다음 행동을 빠르게 판단한다. |
| SSAFY 교육생 | 교육 과정 중 추천/공유되는 채용 정보를 확인한다. | 추천 공고를 놓치지 않고 장바구니에 저장한다. |
| 반복 지원자 | 기본정보, 학력, 경력, 프로젝트, 수상/활동 정보를 여러 지원서에 반복 입력한다. | 서류 입력 정보를 재사용하고 공고별 작성 맥락을 유지한다. |

## 현재 방식의 한계

| 현재 방식 | 한계 |
| --- | --- |
| 채용 플랫폼 북마크 | 공고 저장은 가능하지만 자기소개서 작성 맥락과 참고자료가 분리된다. |
| Notion/Excel 수동 관리 | 자유도는 높지만 공고 저장, 마감 상태, 작성 공간 생성이 자동화되지 않는다. |
| 메모/문서 파일 분산 관리 | JD, 기업 정보, 자소서 초안, 버전이 흩어져 재사용이 어렵다. |
| 브라우저 탭/링크 보관 | 마감일, 지원 상태, 작성 여부를 한눈에 판단하기 어렵다. |

## 핵심 문제

관심 공고와 지원 준비 자료가 분산되어 있어 사용자는 공고별 준비 우선순위와 작성 맥락을 매번 다시 정리해야 한다.

## 문제 근거

| 근거 | 내용 | 연결 요구사항 |
| --- | --- | --- |
| 공고 저장 흐름 분산 | 공고 URL, 마감일, 자소서 문항, 지원 상태를 별도 도구에 옮겨야 한다. | `JOB-001`, `JOB-002`, `EXT-001` |
| 작성 맥락 손실 | 공고별 JD, 참고자료, 자소서 초안, 버전이 한 화면에서 연결되지 않는다. | `WS-001` ~ `WS-004`, `REF-001`, `REF-002` |
| 반복 입력 비용 | 기본정보, 학력, 경력, 프로젝트, 수상/활동을 지원서마다 다시 입력한다. | `PROFILE-001` |
| 추천 활용 어려움 | 추천 공고를 발견해도 바로 저장하고 준비 공간으로 이어지는 흐름이 약하다. | `ONB-002`, `REC-001` |
| 외부 연동 불안정성 | Notion, 확장 프로그램, 외부 사이트 DOM 변화는 실패할 수 있다. | `NOTION-001`, `EXT-001` |

## 해결 방향

- 공고 저장 시 장바구니와 지원 워크스페이스를 함께 생성한다.
- 워크스페이스에서 공고 정보, 참고자료, 도화지, 자소서 버전을 함께 다룬다.
- 서류 입력 정보는 사용자 단위로 저장하고 공고별 작성 공간에서 재사용한다.
- 추천 공고는 별표 저장으로 장바구니와 워크스페이스 흐름에 연결한다.
- 외부 연동 실패는 핵심 DB 저장 흐름을 막지 않도록 분리한다.

## 검증 기준

| 검증 항목 | 확인 방법 | 완료 기준 |
| --- | --- | --- |
| 공고 저장 후 작성 공간이 바로 생기는가 | 확장 프로그램, 추천, 직접 입력 저장 테스트 | 저장된 공고마다 워크스페이스가 생성된다. |
| 사용자가 다음 행동을 판단할 수 있는가 | 대시보드와 장바구니 상태 확인 | 상태 카드와 필터/정렬로 지원 흐름을 찾을 수 있다. |
| 작성 맥락이 유지되는가 | 워크스페이스 화면 검증 | 공고 정보, 참고자료, 도화지, 버전관리를 한 흐름에서 접근한다. |
| 반복 입력을 줄이는가 | 서류 입력 정보 재사용 테스트 | 저장된 기본값이 워크스페이스 입력 보조에 반영된다. |
| 외부 실패가 핵심 저장을 막지 않는가 | Notion/Extension 실패 케이스 테스트 | 핵심 DB 저장은 유지되고 실패 상태가 기록된다. |
